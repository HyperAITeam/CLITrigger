# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

CLITrigger is a full-stack app that automates AI-powered task execution. Users write TODO items in a web UI, and the system spawns isolated git worktrees for each task, running Claude/Gemini/Codex CLI tools in parallel. Built with Express + React + SQLite + WebSocket.

## Commands

```bash
# CLI (npm global install)
npm i -g clitrigger            # global install
clitrigger                     # start server (first run prompts for password)
clitrigger config              # view/change settings (port, password)

# Development (runs server + client concurrently)
npm run dev

# Build
npm run build                  # both client and server (copies client to dist/client/)
npm run build:server           # server only (outputs to dist/server/)
npm run build:client           # client only (outputs to src/client/dist/)

# Production
npm run start                  # serves built app on PORT (default 3000)
npm run start:tunnel           # with Cloudflare Tunnel enabled

# Tests
npm run test                   # all tests (server + client)
npm run test:server            # server tests only (vitest, node env)
npm run test:client            # client tests only (vitest, jsdom env)
npx vitest run src/server/path/to/file.test.ts   # single server test
cd src/client && npx vitest run src/path/to/file.test.tsx  # single client test

# Type checking
npm run typecheck              # server + client

# Desktop (Windows)
build-win.bat                  # EXE + portable (NSIS). --skip-install/-s skips npm ci; --msix switches to appx target with auto-generated self-signed cert
```

## Architecture

### Monorepo Layout

- **`bin/`** — CLI entry point (`clitrigger.js`). Reads `~/.clitrigger/config.json` (creates default `{port:3000, tunnel:true}` on first run), sets env vars, and imports the compiled server. Password is set/changed in the web UI (Settings → Account) — the launcher no longer has a TTY prompt. Legacy `config.password` plaintext (from older installs) is forwarded once via `AUTH_PASSWORD` env so the server can migrate it to a scrypt hash; on the next boot the launcher sees the `.password-migrated` flag and scrubs the plaintext field. Supports `config` subcommand for port/tunnel/path/clear changes (`config password` is deprecated and only prints a redirect to the web UI). Fire-and-forget update check on every launch (npm registry, 5s timeout, silent on failure) prints a one-line `Update available: <v> -> npm i -g clitrigger@latest` hint — does not auto-reinstall or restart. `postuninstall.js` prints cleanup notice on `npm uninstall`.
- **`src/server/`** — Express backend (TypeScript, ESM). Compiled via `tsconfig.server.json` → `dist/server/`.
- **`src/client/`** — React frontend (Vite + TailwindCSS). Has its own `package.json` with separate `npm install`. Dev server proxies API calls to `:3000`. Build output copied to `dist/client/` for npm packaging.
- **`plugin/`** — Hecaton TUI plugin (CommonJS, Deno-compatible). Connects to CLITrigger server as a sidecar client. Built/packaged via `scripts/build-plugin.bat`.

### Server

- **Entry**: `src/server/index.ts` — Express app, middleware, route mounting, graceful shutdown. Auto-retries next port on EADDRINUSE (up to 10 attempts).
- **Database**: `src/server/db/` — SQLite via `better-sqlite3` with WAL mode. Schema uses backward-compatible migrations (adds columns dynamically, never drops tables). 23 tables: `projects`, `todos`, `task_logs`, `schedules`, `schedule_runs`, `cli_models`, `cli_versions`, `plugin_configs`, `discussion_agents`, `discussions`, `discussion_messages`, `discussion_logs`, `sessions`, `session_logs`, `session_raw_chunks`, `session_tags`, `planner_items`, `planner_tags`, `memory_nodes`, `memory_edges`, `memory_logs`, `favorites`, `app_settings`. `projects` carries `vcs_type` (`'git'` | `'svn'` | `null`) + `svn_enabled` (opt-in flag, default 0) + `is_svn_wc` to extend git-only assumptions to SVN working copies; `is_git_repo=1` rows are backfilled to `vcs_type='git'`. Denormalized `todos.total_cost_usd`/`total_tokens` columns enable SQL-only analytics aggregation. `memory_nodes`/`memory_edges` form a per-project knowledge graph with FK CASCADE on `project_id` and edge endpoints; edges carry a `relation_type` (`related`/`precedes`/`example_of`/`counter_example`/`refines`) with UNIQUE(from,to,relation). `memory_logs` records ingest/lint/retrieve/merge events (event_type, severity, source_type/id/title, message, metadata) for the Wiki Activity sub-tab. `session_raw_chunks` stores binary PTY output per session for xterm.js replay with ~2MB rolling cap per session (FK CASCADE). `sessions` carries `memory_inject_mode`/`memory_node_ids`/`memory_raw_file_paths` columns (parity with todos/discussions; `memory_raw_file_paths` is a JSON array of relative paths under `.clitrigger/raw/` for verbatim markdown injection alongside curated nodes) so wiki injection runs at session start. `sessions.tag_id` references the global `session_tags` table for color-coded session grouping; on tag delete, referencing sessions get their `tag_id` explicitly cleared. `session_tags(id, name UNIQUE, color, sort_order, created_at, updated_at)` is a global table for user-defined session category tags (cross-project). `favorites` is a global table (id, name, type, target, args, cwd, icon, sort_order) for user-defined external tool shortcuts (type CHECK: 'executable'|'command'|'url'). `app_settings(key, value, updated_at)` is a global KV table for instance-wide settings (currently `tunnel.name`, `tunnel.hostname`, `auth.password_hash`, `auth.password_changed_at`, `session.default_use_worktree`, `session.default_font_size`). Visual schema at `docs/ERD.md` (auto-generated from `schema.ts` — regenerate manually via `npm run docs:erd` when schema changes).
- **Routes**: `src/server/routes/` — REST endpoints under `/api/`. Auth, projects (including native folder picker via `POST /browse`, `POST /open-folder`, worktree listing/cleanup; `GET /:id` lazily re-runs `isGitRepository(path)` when `is_git_repo` is falsy and persists the result so post-`git init` projects are recognized on next view; analogous lazy `isSvnRepository(path)` runs only when `svn_enabled=1` so non-SVN users never spawn `svn` processes; `GET /:id/git-remotes` returns `[{name, url}]` for the multi-branch push dialog), todos, execution (including `POST /continue` for multi-round), logs (with shared `resolveLocalBaseBranch` fallback for `/todos/:id/diff` and `/todos/:id/result`), images, schedules (including `POST /todos/:id/schedule-on-reset` for rate-limit recovery), plugins, models, cli-status (installation check), analytics (per-project cost/execution stats), sessions (interactive-mode entity, including `tag_id` and `memory_raw_file_paths` writes — both validated server-side, raw paths checked against `.clitrigger/raw/` prefix + traversal), session-tags (global CRUD with `#RRGGBB` hex validation, case-insensitive name uniqueness; on delete, referencing sessions are explicitly cleared), session-settings (`GET/PUT /api/session-settings` over `app_settings` KV — exposes `defaultUseWorktree` for new-session form defaults and `defaultFontSize` (8-28 clamp) for the global terminal font default that broadcasts live to active sessions without an explicit per-session zoom), planner (lightweight task management, plus Markdown `GET /export` / `POST /import` with `text/markdown` body parser for cross-install transfer), review (read-only `GET /api/review/queue`, `/summary`, plus `/diff/:todoId` and `/diff/:todoId/file` for inline diff expansion with worktree → branch ref → master/main fallback chain and structured reason codes `no-branch`/`branch-missing`/`base-branch-missing`; risk classifier server-side; merge/discard/continue actions reuse existing todo endpoints), memory (`GET /api/projects/:id/memory/{graph,nodes,edges}` CRUD + `POST .../preview` for prompt-block preview, `POST /merge` for atomic node merge with edge redirect + wikilink rewrite + tag union, `POST /assets` + `GET /assets/:filename` for body-image upload, `POST /export` for one-way DB→`.clitrigger/wiki/` Markdown dump, `GET /disk-diff` for external-edit detection, `GET /logs` for the Activity sub-tab, `DELETE /raw-files` for raw source cleanup), tunnel (with `GET/PUT /api/tunnel/config` for custom hostname routing — DB-stored in `app_settings`, hostname pattern validation), discussions (with `POST /:id/extract-planner-items` and `/convert-to-planner` for LLM action-item extraction; merge/diff endpoints share the same base-branch fallback as todos), svn (13 endpoints under `/api/projects/:id/svn-*` — status/info/log/diff/commit-files/commit-diff plus add/revert/delete/resolve/commit/update/cleanup; gated on `project.svn_enabled && vcs_type==='svn'` so SVN routes 400 for non-SVN projects, with `revision` regex-validated against svn argument injection and an optional `?wcPath=` query traversal-guarded against the project's `.worktrees` dir for phase-2 multi-working-copy support), debug-logs. Integration routes (Jira, GitHub, Notion, gstack, harness) are mounted via the plugin system.
- **Plugins**: `src/server/plugins/` — Modular integration system. Each plugin (jira, github, notion, gstack, harness) is a self-contained module with its own `PluginManifest`, router, and config. Registered in `index.ts` via `registerPlugin()` and auto-mounted via `mountPluginRoutes()`. Two categories: `external-service` (REST proxy + UI panel) and `execution-hook` (orchestrator pre-execution hook). Config stored in generic `plugin_configs` table (key-value per project+plugin). Legacy `projects` table columns maintained for backward compatibility. The `harness` plugin (`src/server/plugins/harness/`) edits Claude/Gemini/Codex user-config files (`~/.claude/`, `~/.gemini/`, `~/.codex/`) — settings, memory, and MCP servers — via per-CLI adapters with atomic writes (temp + rename), deep-merge to preserve untouched fields, and a path-traversal guard. The Codex adapter parses `~/.codex/config.toml` via `@iarna/toml` and surfaces a `trustLevelMissing` warning when the project isn't in the trusted list. Edits are NOT worktree-scoped (they affect the user's global CLI config), so the client renders an explicit warning banner.
- **Services**: `src/server/services/` — Core business logic:
  - `orchestrator.ts` — Task execution engine. Manages concurrency limits, dependency chains, worktree setup with per-project `use_worktree` toggle plus per-todo tri-state override (`todos.use_worktree`: null=inherit, 0=force-main, 1=force-worktree) resolved via `resolveUseWorktree(project, todo)`. Main-branch todos (effective `use_worktree=false`) are gated in `canStartNow` to run alone — if another todo is active the start is deferred and retried from `startDependentChildren` when a sibling finishes. Also handles CLI invocation, auto-chaining of dependent children, squash merge on dependency completion, dependency chain batch merge (leaf branch → main), CLI fallback on context exhaustion AND on Gemini quota exhaustion (sliding-window detector in `log-streamer.ts`), plugin execution hooks (e.g. gstack skill injection), memory injection via `memory-inject-hook.ts` that prepends a `<long_term_memory>` block to the prompt based on per-todo `memory_inject_mode`/`memory_node_ids`, sandbox mode (strict: directory-scoped permissions even for non-worktree runs, permissive: full access), stale process liveness checker (30s interval), multi-round continue (`continueTodo()` reuses existing worktree with `--continue` flag), denormalized cost/token persistence on both success and failure paths for SQL-only analytics, and best-effort `captureReviewMetadata(todoId)` invocation on both success and failure paths to populate review-queue summary/diff stats.
  - `memory-injector.ts` — Builds the `<long_term_memory>` prompt block from a per-project knowledge graph (`memory_nodes` + `memory_edges`). Serializes node body and adjacent edges using arrow-notation (e.g. `Node A —(precedes)→ Node B`). CLI-agnostic — output is plain text prepended to the prompt, so Claude/Gemini/Codex all see identical context with no adapter changes. Four injection modes: `none`, `all`, `selected` (specific node IDs), `auto` (LLM-picked candidates per call). Token-safety: `buildNodeSummary` caps lines per node and falls back to title-only listings when the budget is exceeded; System nodes are excluded from `all`. Also exports `buildRawFileBlock(projectRoot, paths, opts?)` — verbatim injection of `.clitrigger/raw/<path>` markdown sources alongside curated nodes, wrapped in a `<raw_source_files>` XML block. Per-file 50KB char cap with `[... truncated]` marker; `path.resolve` traversal guard ensures inputs stay under the raw dir (skip reasons: `outside-raw-dir` / `missing` / `read-error`). Selector is orthogonal to mode — `mode='none'` with non-empty `memory_raw_file_paths` still injects the raw block.
  - `memory-retriever.ts` — Backs the `auto` injection mode. Extracts candidates from a project's wiki (title/body/tags) and runs a one-shot headless LLM call to score and select the relevant subset given a query (todo/discussion text, or session title+description). Cost is one extra retrieval call but the main prompt stays compact.
  - `wiki-exporter.ts` — One-way DB → `.clitrigger/wiki/<entity>/<slug>.md` Markdown export with YAML frontmatter (id, tags, edges, source_path). Backs the Wiki tab's Export button and Disk diff/Rebuild flow. Truth source stays in the DB; disk content can live in git or Obsidian but is overwritten by Rebuild.
  - `wiki-index.ts` — Auto-maintains "System" nodes (Schema + Index) shown in a separate sidebar section with an "edits are overwritten" banner. System nodes are excluded from entry groups, retrieval candidates, lint, and `all`-mode injection.
  - `memory-ingest.ts` — Source → wiki ingest pipeline. Splits long sources (>8000 chars) at paragraph boundaries into sequential chunks (max 4 chunks of 7000) and re-fetches nodes between chunks so dedup sees earlier-chunk additions. Tees stdin/stdout/stderr to `debugLogger` (synthetic `mem-{kind}-{sourceType}-{sourceId|ts}` todoId) when `project.debug_logging` is on. Returns a skip breakdown (parse fail, dup, unique, bad id/edge) for the result modal. Auto-ingest results broadcast via `memory:ingest-finished` WS event for the sidebar toast.
  - `memory-inject-hook.ts` — Wraps `memory-injector.ts` for orchestrator/session-manager use. Logs the injection mode and node count to `task_logs`/`discussion_logs`/`session_logs` for visibility. Dispatches to `memory-retriever.ts` for `auto` mode. Builds the curated-node block and raw-file block independently and concatenates them with a blank line between (single prepend); raw-file skip reasons are surfaced per-file in the same logs. Gate condition: `mode !== 'none' || rawFilePaths.length > 0` so raw-only injection works.
  - `discussion-extractor.ts` — Headless one-shot LLM call that converts a completed discussion's transcript into curated planner items. Uses `runHeadless(cliTool, prompt)` with stdin-piped prompt (bypassing Windows cmd.exe argument-length limit), respects `project.cli_tool` (claude/gemini/codex), wraps via `cmd.exe /c` on Windows, and applies a 120s timeout. Parses a tolerant JSON array of `{title, description, priority}` from the model output (ignores code-fence wrapping and surrounding prose). Powers `POST /api/discussions/:id/extract-planner-items` (preview) and `POST /api/discussions/:id/convert-to-planner` (persist).
  - `review-capture.ts` — Best-effort capture of review-queue metadata for finished todos. `pickSummaryFromLogs(todoId)` selects the latest assistant message (or non-empty plain output as fallback) from the most recent round and trims to 240 chars. `computeDiffStats(worktreePath, defaultBranch)` parses `git diff <branch>...HEAD --shortstat` for files/lines. Persists to `todos.summary`/`diff_files`/`diff_lines`. All failures are swallowed so review capture never blocks orchestration.
  - `claude-manager.ts` — Spawns/manages child processes (node-pty for any interactive mode and TTY-requiring tools like Codex, child_process for headless/verbose). Windows cmd.exe wrapper for .cmd shims. Interactive mode uses PTY with stdin wrapped as Writable for WebSocket relay. Uses each adapter's `stdinSubmitSequence` (`\r` for Claude/Codex, `\r\n` for Gemini) for every PTY write path (initial prompt delivery, interactive relay, delayStdin fallback). Adapter-driven `autoRespondRules` handle CLI-specific dialogs (Claude workspace trust, Gemini "Trust folder" + update prompts) without CLI-specific hardcoding; rules flagged `blocking` gate the initial prompt until the dialog clears.
  - `cli-adapters.ts` — Adapter pattern abstracting Claude/Gemini/Codex CLI differences (args, stdin format, output format, session resume, interactive mode). Each adapter declares `supportsInteractive`, `stdinSubmitSequence`, `delayStdinUntilReady`, `readyIndicatorPattern`, and `autoRespondRules[]` (blocking/non-blocking). Supports `SandboxMode` (strict/permissive) per CLI tool. Codex's `continueSession` emits `exec resume --last`; Gemini's emits `--resume latest`. In interactive mode Codex skips the `exec` subcommand to launch the top-level TUI.
  - `log-streamer.ts` — Streams stdout/stderr to DB. Two modes: JSON lines (Claude structured output) and plain text (Gemini/Codex). Claude stream-json events are classified into `assistant`/`tool_use`/`tool_result` types (not flattened to `output`) so the client can render conversational views. Parses token usage, commit hashes, and Claude's `rate_limit_event` (broadcast via WebSocket for reset-time recovery). Detects context exhaustion for CLI fallback chain. Also detects Gemini quota exhaustion via a sliding-window counter (`QUOTA_EXHAUSTION_PATTERN` matching `exhausted your capacity` / `quota will reset`); 3 hits within 60s trips a `quotaKillCallback` (wired by orchestrator to `claudeManager.stopClaude`) so the existing exit-code → `getNextFallbackCli` path takes over — needed because Gemini's CLI retries internally without exiting. Plain-text streams (Gemini/Codex) are gated through `pty-output-filter.isPlainTextNoise(line, state)` with a per-todo `noiseFilterMap` so block state survives across stdout/stderr splits, and the map is cleaned up unconditionally in `getTokenUsage`. Supports round-based log tagging via `setRound()` for multi-round continue.
  - `worktree-manager.ts` — Git worktree lifecycle via `simple-git`. Branch name sanitization (Korean → slug, `feature/` prefix, 40 char max, duplicate suffix `-2`/`-3`). `createWorktree(..., autoInstall)` runs `npm install` in the worktree only when the caller opts in via `project.npm_auto_install` (default OFF); Todo/Discussion/Session orchestrators all pass the same flag through, so the worktree action stays ecosystem-neutral for non-npm projects. Auto-adds `.worktrees` to project `.gitignore`. Squash merge auto-aborts on conflict to prevent dirty worktree state. Also provides 16 Git action methods (stage, unstage, commit, pull, push, fetch, branch, checkout, merge, stash, discard, tag, diff) and commit inspection methods (`getCommitFiles`, `getCommitDiff` with root/merge commit handling) for the web Git client. `gitPush(dir, opts)` takes an object (`{remote, branches: PushBranchSpec[], pushAllTags, force}`) so the SourceTree-style multi-branch push dialog can push any combination of `local:remote` refspecs in one call; `--force` always becomes `--force-with-lease` to prevent stale-view overwrites, and rows with `setUpstream:true` split off into a second `git push -u` invocation (with `--tags` stripped on the second call to avoid invalid combos in some git versions). `getRemotes(dir)` returns `[{name, url}]` for the dialog's remote select. All `simpleGit(...)` call sites go through the `createGit()` factory in `src/server/lib/git.ts`, which sets `core.quotePath=false` per invocation so non-ASCII filenames (Korean, CJK, emoji) are not C-escaped in diff/status output. The same module exports `resolveLocalBaseBranch(git, configured)` — a shared helper that returns the configured `default_branch` if it exists, falls back to `master`/`main`, otherwise null; consumed by `routes/review.ts`, `routes/logs.ts`, and `routes/discussions.ts` so all diff/merge endpoints share consistent base-branch resolution.
  - `scheduler.ts` — Cron (recurring) and one-time schedules via `node-cron`.
  - `skill-injector.ts` — Injects gstack skill files into `.claude/skills/` in worktrees (Claude CLI only). Used by gstack plugin's `onBeforeExecution` hook.
  - `discussion-orchestrator.ts` — Multi-agent discussion engine. Round-based turn execution where agents speak sequentially, each receiving the full discussion history in their prompt. Supports start/stop/pause/resume, user message injection, turn skipping, auto-implement (automatically triggers implementation round on discussion completion), a special implementation round (max_rounds+1) where a designated agent writes code, and a per-agent `can_implement` flag that lets flagged agents commit on their regular turn (prompt is softened to "prototype the smallest slice" and `project.default_max_turns` is granted in full, not the 10-turn discussion cap). Commits chain naturally on the shared worktree/branch, so the final implementation round keeps its "finish what's missing, commit everything" role. Uses worktree isolation and sandbox mode like todos. Memory injection (`memory_inject_mode`/`memory_node_ids`) prepends the same `<long_term_memory>` block as the todo orchestrator before each agent turn.
  - `session-manager.ts` — Interactive session engine (sibling of `orchestrator`/`discussion-orchestrator`). Wraps `claudeManager` + `logStreamer` to run a single long-lived interactive CLI session per `sessions` row. Supports worktree isolation (opt-in per session), stale-session recovery on startup, `session_logs` retention cleanup, and raw binary PTY output streaming to `session_raw_chunks` table (batched 4KB-or-100ms DB flush, ~2MB rolling cap per session) for xterm.js pixel-perfect playback. Client subscribes to raw bytes via WebSocket; **DB chunks are the single source of truth** — `session:subscribe` calls `flushPendingRaw(sessionId)` (public variant of the in-flight buffer drain) before sending only the persisted chunks (no ring-tail replay), so the welcome banner doesn't get rendered twice. Accepts optional `ptyCols`/`ptyRows` at session start to spawn PTY at actual viewport dimensions. Wiki injection runs via `applyMemoryInjection()` (same as orchestrators) using `${title}\n${description}` as the auto-mode retrieval query. Initial prompts are **stashed in an in-memory `pendingInitialPrompts` map** rather than auto-written to the PTY; clients fetch via `GET /sessions/:id/pending-prompt`, then `POST /submit-initial` (write to PTY) or `/skip-initial` (discard). Pending entries are cleared on stop/exit/error so a stuck panel can't outlive its session. Also maintains a per-session **`startupInputBuffer` Map** so PTY-bound keystrokes that arrive during the spawn window (between status='running' flip and `process_pid` populated) are queued instead of dropped — `writeTerminalInput(sessionId, input)` pushes to the queue when present, and the queue is drained synchronously into PTY stdin at process_pid update time so type-ahead lands in the right order. Buffer entries are cleaned up on stop/exit/spawn-failure paths. Backs both the Sessions tab (with floating window UI) and its `/cleanup` endpoint.
  - `pty-output-filter.ts` — Noise filter for PTY and headless plain-text streams. Line-level patterns drop CLI spinner frames (including concatenated/bare-\r redraws), status bars, thinking animations, TUI menu indicators (☰ ○ ⏵), worktree path echo, and numbered prompt echo (`3> ...`); cursor sequences are replaced with spaces so word boundaries survive. A stateful block filter (`PtyFilterState.activeBlock` + `blockLineCount`) absorbs multi-line noise: xterm.js parser-state dumps (until outer `}` at column 0) and conpty/node-pty stack traces (until `Node.js vX.X.X`), with a 200-line runaway guard. `isNoiseLine` drops AttachConsole/conpty/xterm.js lines BEFORE the keep-signals `/^Error/` exception so they cannot leak. The `isPlainTextNoise(line, state)` export is consumed by `log-streamer.ts` so the same cleanup applies to headless Gemini/Codex output, not just interactive PTY.
  - `cli-status.ts` — CLI tool installation checker. Runs `--version` on Claude/Gemini/Codex in parallel with 60s caching. Backs `GET /api/cli/status` and surfaces install guidance in the project settings panel. Fire-and-forget triggers `model-sync.ts` on each status check.
  - `model-sync.ts` — CLI model list synchronizer. Probes each CLI's `--help` for supported models and merges with a fallback registry at `src/server/data/cli-models-registry.json` (probe-only loses coverage when CLI help format changes; registry-only misses new models). Flags deprecated entries in `cli_models` and writes the CLI version to `cli_versions`.
  - `debug-logger.ts` — CLI debug logging service. When `project.debug_logging` is enabled, captures raw stdin/stdout/stderr to `.debug-logs/` via PassThrough stream tee (non-invasive to existing log stream). Auto-cleans old logs on startup based on `LOG_RETENTION_DAYS`. Auto-adds `.debug-logs` to project `.gitignore`.
  - `prompt-guard.ts` — Prompt injection detection and sanitization for external inputs (Notion/GitHub/Jira imports).
  - `tunnel-manager.ts` — Cloudflare Tunnel management via `cloudflared` subprocess. Uses bundled npm `cloudflared` package binary as primary source, falls back to system PATH and Windows-specific paths.
  - `svn-manager.ts` — SVN working-copy manager. Mirrors the parts of `worktreeManager` that the GUI consumes (status, info, log, diff, commit-files/diff, plus add/revert/delete/resolve/commit/update/cleanup) so `DiffViewer`/`CommitFileList` render SVN data with no per-VCS branching at the leaf level. Status entries always have `index === ' '` (SVN has no staging area) and the `working_dir` slot carries the SVN status char, matching git porcelain shape; log revision numbers go into the `hash` slot as strings. XML parsing uses regex against svn's stable `--xml` schema (no XML dependency). `lib/svn.ts` provides the underlying `runSvn` helper with UTF-8 env (`LC_ALL=en_US.UTF-8`) and `--non-interactive` to prevent credential-prompt hangs, and `isSvnRepository(dir)` (`.svn/` stat + `svn info` fallback). `svn-enabled` projects force `orchestrator.getMaxConcurrent` to `1` since SVN has no worktree-equivalent isolation in phase 1.
  - `clipboard-writer.ts` — Pushes raw image bytes into the host OS clipboard so a CLI subprocess (Claude/Codex/Gemini) can pick them up via its native Alt+V handler — no disk file ever touches the user's project tree. Platform writers: Windows (PowerShell stdin → `MemoryStream` → `System.Windows.Forms.Clipboard.SetImage`, in-memory), macOS (brief tmp file in `os.tmpdir()` + `osascript`, unlinked immediately), Linux (`wl-copy` stdin when `WAYLAND_DISPLAY` is set, else `xclip` — fallback in either order). Concurrent calls serialized via a process-wide promise-chain mutex so two sessions don't clobber each other's bytes before the CLI reads them. 10s spawn timeout. Backs `POST /api/sessions/:id/paste-image`.
- **WebSocket**: `src/server/websocket/` — Real-time log streaming, binary PTY frames, and status broadcasts. Session-authenticated. Supports stdin relay for interactive mode and per-client binary subscription management (session:subscribe/unsubscribe for xterm.js replay, session:terminal-input/session:resize for keystroke/resize relay).
- **Auth**: Session-based (`express-session`). Password is stored as a `node:crypto.scrypt` hash (`scrypt$N=...,r=...,p=...$salt$hash` format) in `app_settings.auth.password_hash`; `auth.password_changed_at` is bumped on every change so the auth middleware can invalidate sessions issued before that timestamp. Skips `/api/auth/*` and `/api/health`. **Setup mode** kicks in when no hash exists and `DISABLE_AUTH !== 'true'` — server still boots but the Cloudflare tunnel auto-start is held until `POST /api/auth/setup` completes (prevents external access from grabbing initial setup). Legacy `AUTH_PASSWORD` env (plaintext) is migrated to a hash on first boot then deleted from memory. Routes: `POST /api/auth/login`, `POST /api/auth/setup` (one-shot, 409 once initialized), `PUT /api/auth/password` (oldPassword verify, keeps own session alive), `POST /api/auth/logout`, `GET /api/auth/status` (`{authenticated, authRequired, setupRequired}`). Auth middleware fully disabled when `DISABLE_AUTH=true` (plugin/headless mode).

### Key Patterns

- **CLI Adapter Pattern**: All CLI tool differences are isolated in `cli-adapters.ts`. Adding a new CLI means implementing the `CliAdapter` interface.
- **Integration Plugin Pattern**: External service integrations (Jira, GitHub, Notion) and execution hooks (gstack) are self-contained plugins in `src/server/plugins/` and `src/client/src/plugins/`. Each plugin exports a `PluginManifest` (server) and `ClientPluginManifest` (client). Adding a new integration means: create a plugin directory, implement the manifest, and call `registerPlugin()` — no core code changes needed. Config stored in `plugin_configs` table (generic key-value). Two plugin categories: `external-service` (REST proxy routes + panel tab) and `execution-hook` (pre-execution hook in orchestrator).
- **Worktree Isolation**: Each task gets its own git worktree in `.worktrees/`. Child tasks can inherit parent worktrees. Per-project `use_worktree` toggle (default: on) allows running directly on main branch without worktree overhead; when disabled, server forces `max_concurrent=1` to prevent conflicts. Per-todo tri-state override (`todos.use_worktree`: null=inherit, 0=force-main, 1=force-worktree) lets individual tasks opt in/out; main-branch todos are gated to run alone via `canStartNow` and retried via the dependent-children flow when siblings finish.
- **Graceful Shutdown**: Server handles SIGTERM/SIGINT — kills running CLI processes, stops scheduler, closes tunnel. Also shuts down on stdin EOF (plugin sidecar mode).
- **Headless Mode**: `HEADLESS=true` skips static file serving (API-only mode for plugin/embedded use). `DISABLE_AUTH=true` removes auth middleware (local-only plugin scenarios).
- **DB Migrations**: Schema changes add columns with `ALTER TABLE ... ADD COLUMN` guarded by try/catch, so the app works with both old and new DB files. Plugin configs use a separate `plugin_configs` table with automatic migration from legacy project columns.
- **Sandbox Mode**: Per-project `sandbox_mode` (strict/permissive). Strict mode uses each CLI's native sandboxing to restrict file access to the worktree directory. Claude: auto-generated `.claude/settings.json` with absolute-path patterns (`${workDir}/**`) for Read/Edit/Write — relative paths like `./` are ineffective because Claude resolves paths to absolute internally. The work-dir is normalized to forward slashes (`workDir.replace(/\\/g, '/')`) before substitution so Windows backslash paths don't break Claude's path-normalizing permission matcher (mixed-separator patterns silently rejected every Edit/Write match and forced a Bash fallback). Codex: `--full-auto` + `--add-dir .git`; Gemini: prompt-level path restriction.
- **Agent Discussion**: Multiple AI agents with different roles (architect, developer, reviewer, etc.) discuss a feature in rounds before implementation. Each agent speaks sequentially with full history context. After discussion completes, a designated agent implements the consensus, OR users can run `discussion-extractor.ts` to convert the transcript into curated planner items via a one-shot headless LLM call (preview + edit + persist flow, with `source_discussion_id` linking back to the originating discussion). Uses same worktree isolation and CLI adapter patterns as todos.
- **Wiki Injection (Karpathy LLM-Wiki pattern)**: Per-project knowledge graph (`memory_nodes` + `memory_edges`) is curated in the Wiki tab and selectively prepended to todo/discussion/session prompts as a `<long_term_memory>` block (XML tag kept verbatim — LLMs treat the literal phrase as a semantic hint). Four modes: `none` (default), `all`, `selected` (specific entry IDs), `auto` (one-shot headless LLM retrieval per call via `memory-retriever.ts`, picks candidates from title/body/tags). The block is plain text, so it works identically across Claude/Gemini/Codex with no adapter changes — `memory-injector.ts` builds it, `memory-inject-hook.ts` wraps with logging, and the orchestrators (and session-manager) prepend it before the existing plugin hook loop. Edges carry typed relations (`related`/`precedes`/`example_of`/`counter_example`/`refines`); the client cycle-guards `precedes`/`refines` to avoid serialization loops. Token-safety: per-node line cap with title-only fallback; lint chunks large wikis; preview modal shows char/token estimates. **Raw markdown injection** (orthogonal to mode): `todos/discussions/sessions.memory_raw_file_paths` (JSON array) selects original `.clitrigger/raw/*.md` sources to inject verbatim alongside curated nodes inside a parallel `<raw_source_files>` block — useful when curation dropped detail. Path-traversal-guarded against the raw dir, 50KB per-file cap with truncation marker, skip reasons surfaced in execution logs. Selector is mode-independent: `mode='none'` with non-empty raw paths still triggers injection. Maintenance loop: lint surfaces duplicates/orphans/stale with inline fix actions (merge endpoint redirects edges + cascades wikilink rewrites + unions tags atomically). Activity sub-tab (`memory_logs` table) records ingest/lint/retrieve/merge events. Portability: `wiki-exporter.ts` does one-way DB → `.clitrigger/wiki/<entity>/<slug>.md` Markdown export with YAML frontmatter; Disk diff surfaces external edits; Rebuild overwrites disk from DB. System nodes (Schema + Index, auto-maintained by `wiki-index.ts`) live in a separate sidebar section with an "edits are overwritten" banner and are excluded from entry groups, lint, retrieval candidates, and `all`-mode injection. Note: the UI was renamed from "Memory" / "장기기억" to "Wiki" / "위키", but DB tables, API routes (`/api/projects/:id/memory/*`), TypeScript identifiers, file names (`MemoryList.tsx`, `memory-injector.ts`), and the `<long_term_memory>` prompt tag intentionally retain the original "memory" naming to avoid migration risk and preserve the LLM semantic signal.
- **Floating Window + Group Pattern**: `SessionWindow` renders via `createPortal(..., document.body)` so groups survive tab switches and don't get destroyed when ProjectDetail re-mounts. State is managed by `SessionWindowsHost` as a layout tree (`stack` | `split` nodes) per `OpenGroup`, persisted to `sessionGroups:{projectId}` localStorage and gated on `sessions` having loaded so an empty list during loading doesn't nuke restored state. Drag, resize, and split mutate `style.transform` / `flex-basis` directly during gestures (bypass React rerenders, keep xterm canvas stable) and commit to React state on mouseup. Viewport clamping keeps the titlebar ≥80px visible. Aero edge snap (8px from viewport edges) + neighbor sticky snap (10px) live preview via portal during drag. VS Code-style docking: a 5-zone diamond (top/bottom/left/right/center) hit-tests via `document.elementFromPoint` against `data-group-id`/`data-stack-path`-tagged stacks; center drop merges tabs, side drop creates a split sibling. Eager tab tearing detaches a tab into a floating group when it moves ≥12px from the source rect. Single-stack chrome drag detects docks by setting `pointer-events: none` on the wrapper for the gesture (so the elementFromPoint lookup finds destination stacks instead of self-hitting). All tabs in a group stay mounted (`display` toggled) so xterm state and live PTY output never drop on tab switches. Mobile (<768px) collapses to fullscreen of the active tab only, no docking widgets. Apply this pattern for any persistent multi-pane floating UI.
- **Raw Binary Streaming for xterm.js**: Sessions stream raw PTY bytes (not stripped/processed) to enable pixel-perfect terminal rendering. Server maintains a 256KB per-pid ring buffer in `claudeManager` and a ~2MB rolling-cap `session_raw_chunks` DB table for cross-restart replay. **DB chunks are the single source of truth for replay** — `session:subscribe` calls `flushPendingRaw(sessionId)` (drains the in-flight buffer to DB synchronously) before sending only the persisted chunks, with no ring-tail replay (the ring is a strict subset of DB and replaying both produced duplicate welcome banners). Live stream is delivered via the same subscription thereafter. Binary frame parsing bypasses React state (direct per-sessionId callback Map in useWebSocket) for performance. xterm.js receives raw writes; ANSI colors, cursor control, and TUI styling all work identically to native terminal.
- **Tunnel Custom Hostname**: Cloudflare named tunnels can be routed through a user's own domain to inherit its reputation (avoids the "dangerous site" warnings on `*.trycloudflare.com` / `*.cfargotunnel.com`). Server stores `tunnel.name` and `tunnel.hostname` in the global `app_settings` KV table (DB-first, env fallback `TUNNEL_NAME`/`TUNNEL_HOSTNAME`). `tunnel-manager.ts` keeps cloudflared CLI args unchanged — the custom hostname is purely a display/config layer; the user runs `cloudflared tunnel route dns <name> <hostname>` themselves. Validation: hostname pattern regex, rejects `localhost` / spaces / non-domain strings, requires `tunnelName` when `customHostname` is set. CLI: `clitrigger config tunnel hostname <host>|clear`.
- **Opt-in VCS Pattern**: SVN support exists alongside git but is gated by `project.svn_enabled` (default 0) so non-SVN users never spawn `svn` processes, see SVN routes, or have the panel/banner UI rendered. Detection (`isSvnRepository`) is run lazily — only when `svn_enabled` flips on, or when the project is loaded with the flag already set but `vcs_type` empty. The same shape extends to future VCS additions (Mercurial, Fossil): add a `vcs_type` value, an opt-in flag column on `projects`, and a normalizer that maps the VCS's status output to git porcelain shape (`{path, index, working_dir}`) so `DiffViewer`/`CommitFileList` work unchanged. `cli-status.ts` carries an `isVcs` flag on tool entries to skip `model-sync.ts` for non-model-bearing CLIs.
- **Native Focus Handoff (Electron)**: Renderer `element.focus()` and React `autoFocus` only move DOM focus — they don't reclaim the native HWND keyboard focus when it's been captured by an embedded native view (e.g., xterm.js's helper textarea after a session). The fix is a main-process IPC bridge: preload exposes `electronAPI.imeReset` → `ipcRenderer.send('ime:reset')` → `mainWindow.webContents.focus()` in main. Renderer code that mounts after such a capture (e.g., `SessionForm`) calls `imeReset()`, RAFs twice to let xterm's own focus restoration settle, then focuses the target input. Same pattern applies to lock-screen/screensaver resume: `mainWindow.on('focus', () => webContents.focus())` re-routes OS-level focus on every activation.
- **Electron Auto-Update**: Packaged desktop builds use `electron-updater` against GitHub Releases (`latest.yml` + blockmap already published by `release.yml --publish always`). `electron/main.cjs::setupAutoUpdater()` runs only when `app.isPackaged`, kicks the first check 5s after window creation (non-blocking), auto-downloads in the background, and prompts "지금 재시작 / 나중에" on `update-downloaded`; defer falls back to `autoInstallOnAppQuit`. Help menu has a manual "업데이트 확인" item with an `updateCheckInFlight` guard. No release-workflow change needed.
- **Failure Tolerance**: On startup, stale "running" todos are reset to "failed", stale "running" discussions are reset to "paused". Plugin execution hook failures are logged but don't block CLI execution.
- **npm CLI Packaging**: Published as `clitrigger` on npm. `bin/clitrigger.js` handles first-run setup, reads config from `~/.clitrigger/config.json`, sets `PORT`/`AUTH_PASSWORD`/`DB_PATH` env vars, then dynamically imports `dist/server/index.js`. Non-blocking update check on every launch prints an upgrade hint; installation is left to the user. Build copies client output to `dist/client/`; server resolves static files from `../client` (npm install) or `../../src/client/dist` (dev) via fallback. Cloudflared bundled as npm dependency for zero-config tunnel.

## Environment

Config via `.env` (see `.env.example`) or `~/.clitrigger/config.json` (npm global install) / Electron `userData/config.json`. Key vars: `AUTH_PASSWORD` (optional — only used as a one-shot migration source for the scrypt hash on first boot; web UI Setup screen is the canonical entry), `PORT` (default 3000), `DB_PATH` (database location), `TUNNEL_ENABLED`, `TUNNEL_NAME` (named tunnel), `TUNNEL_HOSTNAME` (custom domain for named tunnel), `LOG_RETENTION_DAYS`, `HEADLESS` (skip frontend serving), `DISABLE_AUTH` (skip auth middleware). DB-stored settings (via `app_settings` KV table) take precedence over env vars for tunnel name/hostname; the password hash itself lives only in `app_settings.auth.password_hash`.

## UI Guidelines

> 시각 디자인 결정(색상, 타이포그래피, spacing, 컴포넌트 스타일)은 `.claude/docs/design.md`를 기준 레퍼런스로 참조할 것.

### Floating elements must render via portal

Tooltips, dropdowns, popovers, hover help boxes, context menus, and "more" menus MUST be rendered through `createPortal(..., document.body)` with `position: fixed` + viewport clamping (using the anchor element's `getBoundingClientRect`). Do NOT rely on `position: absolute` inside the component tree — header cards, list rows, badge rows, and worktree/todo containers all have `overflow`, `border-radius`, or `transform` ancestors that will clip the floating element.

Past bugs from this exact mistake: project header deprecated-model tooltip was clipped by the header card border; todo-item "more" menu (`⋮`) was clipped by the task row. The portal + fixed + clamp pattern lives in `TodoItem.tsx`'s `MoreMenu` and `ProjectHeader.tsx`'s `DeprecatedModelBadge` — copy that pattern for any new floating UI.

Checklist for every floating element:
- Render into `document.body` via `createPortal`
- Use `position: fixed` with top/left computed from the anchor's `getBoundingClientRect()`
- Clamp within viewport: flip above on vertical overflow, shift left on horizontal overflow, keep ≥8px from edges
- Recompute on `scroll` (capture phase) and `resize`
- Use `z-tooltip` z-index class

## Task Execution Guidelines

When working on tasks in this repository (especially via CLITrigger worktrees), follow these efficiency rules:

### Efficiency
- Use grep/glob to find relevant files FIRST. Do NOT read files one by one to explore the codebase.
- Only read files you intend to modify or that are directly needed to understand the change.
- Do NOT launch Agent/Explore subagents for simple, targeted tasks (e.g., CSS changes, config updates, single-file fixes). Use direct grep → read → edit.
- Make all related edits in a single pass. Do not re-read files you already read.
- Prefer `replace_all: true` for repetitive changes across a file.
- Aim for under 15 tool calls for simple tasks, under 30 for complex ones.

### Completion
- Once the task is complete, commit and stop immediately.
- Do not perform additional refactoring, optimization, or testing beyond what was explicitly requested.
- Do not add comments, docstrings, or type annotations to unchanged code.
- Do not review your own changes unless asked.
